# Project Description

## Название проекта

Interactive Animated Sprite Demo

## Краткое описание

Проект представляет собой браузерное TypeScript-приложение с интерактивной анимацией, реализованной с помощью игрового движка Phaser.

На экране отображается игровая область размером 1000×600 пикселей с фоновым изображением. В центре экрана находится анимированный объект размером не менее 400×400 пикселей. Анимация состоит минимум из 5 кадров и проигрывается циклически.

При клике или касании экрана анимированный объект плавно перемещается в точку взаимодействия. Движение должно быть визуально плавным, а траектория по возможности нелинейной.

Проект должен корректно работать в десктопных и мобильных браузерах, поддерживать масштабирование, центрирование игрового поля, обработку изменения размера окна, работу в полноэкранном режиме и типовые пользовательские сценарии.

## Цель проекта

Цель проекта — продемонстрировать умение разрабатывать небольшое интерактивное браузерное приложение на TypeScript с использованием игрового движка, объектно-ориентированной архитектуры и корректной обработкой пользовательского ввода.

Проект также должен показать понимание:

* структуры игрового приложения;
* работы с игровым циклом;
* загрузки и отображения ассетов;
* кадровой анимации;
* обработки mouse и touch input;
* масштабирования canvas;
* fullscreen API и fallback-поведения;
* разделения ответственности между классами;
* выноса настраиваемых параметров в конфигурационные файлы.

## Основной пользовательский сценарий

1. Пользователь открывает приложение в браузере.
2. Приложение загружает фон и анимированный объект.
3. Анимированный объект появляется в центре игрового поля.
4. Анимация объекта автоматически воспроизводится по кругу.
5. Пользователь кликает мышью или касается экрана.
6. Объект плавно перемещается в выбранную точку.
7. Если пользователь быстро нажимает в другое место, текущее движение отменяется и объект начинает движение к новой цели.
8. При изменении размера окна игровая область масштабируется и остаётся центрированной.
9. На мобильном устройстве приложение пытается перейти в полноэкранный режим после первого пользовательского жеста.
10. Если настоящий fullscreen недоступен, используется fallback-режим с максимальным занятием viewport.

## Логическое игровое пространство

Внутреннее игровое пространство имеет фиксированный размер:

* ширина: 1000 px;
* высота: 600 px.

Все игровые координаты рассчитываются относительно этого пространства.

Масштабирование не изменяет координатную систему игры. Оно влияет только на визуальный размер canvas в браузере.

## Визуальные элементы

### Фон

Фоновое изображение занимает всю игровую область:

* размер: 1000×600;
* позиция: левый верхний угол;
* масштабирование: соответствует размеру игрового пространства.

### Анимированный объект

Анимированный объект находится поверх фона.

Минимальные требования:

* размер: 400×400 px;
* количество кадров: минимум 5;
* начальная позиция: центр игрового поля;
* анимация: циклическая;
* управление: перемещение по клику или касанию.

## Поведение движения

При пользовательском вводе объект должен перемещаться к выбранной точке.

Требования к движению:

* движение должно быть плавным;
* приоритет имеет последнее действие пользователя;
* объект не должен выходить за границы игрового поля;
* если точка находится слишком близко к границе, цель движения должна быть скорректирована;
* траектория движения должна быть по возможности нелинейной;
* движение должно корректно работать при изменении масштаба окна.

Для нелинейной траектории используется квадратичная кривая Безье.

## Обработка пользовательского ввода

Приложение должно поддерживать:

* клик мышью;
* touch input;
* несколько одновременных касаний;
* быстрые повторные клики;
* взаимодействие после resize;
* взаимодействие после перехода в fullscreen и выхода из него.

Основная стратегия обработки нескольких указателей:

* последнее действие пользователя имеет приоритет;
* текущее движение отменяется;
* объект начинает движение к новой цели из текущей позиции.

## Масштабирование

На десктопе приложение масштабируется до максимального размера:

* 1280×768 px.

Коэффициент масштабирования вычисляется по меньшей стороне окна.

Игровая область должна быть центрирована по горизонтали и вертикали.

На мобильных устройствах приложение масштабируется до максимально возможного размера, сохраняя пропорции игрового поля.

## Fullscreen

На мобильных устройствах приложение должно попытаться перейти в полноэкранный режим после первого клика, касания или свайпа.

Если браузер не поддерживает fullscreen для canvas или HTML-элементов, приложение должно использовать fallback-режим:

* canvas занимает максимально доступную область viewport;
* страница не должна скроллиться;
* игровая область остаётся центрированной;
* пользовательский ввод продолжает работать корректно.

## Архитектурный подход

Проект должен быть реализован с использованием принципов объектно-ориентированного программирования.

Логика не должна находиться целиком в одной сцене. Ответственность должна быть разделена между классами.

Основные сущности проекта:

* GameApplication — запуск Phaser-приложения;
* PreloadScene — загрузка ассетов;
* MainScene — создание основной сцены и связывание объектов;
* AnimatedActor — анимированный игровой объект;
* InputController — обработка mouse и touch input;
* MovementController — управление перемещением объекта;
* TrajectoryFactory — построение нелинейной траектории;
* BoundsService — ограничение движения в пределах игрового поля;
* ViewportController — обработка resize и scaling;
* FullscreenController — управление fullscreen и fallback-режимом.

## Конфигурация

Все изменяемые параметры должны быть вынесены в конфигурационные файлы.

К конфигурируемым параметрам относятся:

* размер игрового поля;
* максимальный desktop-размер;
* параметры масштабирования;
* пути к ассетам;
* размеры кадров анимации;
* количество кадров;
* frame rate;
* скорость движения;
* минимальная и максимальная длительность движения;
* коэффициент изгиба траектории;
* стратегия обработки нескольких касаний;
* настройки fullscreen;
* настройки legacy-совместимости.

## Ограничения

Проект должен учитывать поддержку старых браузеров, включая IE11.

Так как современные версии браузеров и игровых библиотек могут иметь ограничения по legacy-совместимости, перед основной разработкой необходимо выполнить compatibility spike:

* собрать минимальное Phaser-приложение;
* проверить запуск в целевых браузерах;
* проверить необходимость polyfills;
* убедиться, что выбранная версия Phaser подходит под требования.

Также необходимо учитывать, что настоящий fullscreen может быть недоступен в некоторых мобильных браузерах, особенно на iPhone Safari. В таких случаях должен использоваться fallback-режим.

## Критерии готовности

Проект считается готовым, если:

1. Приложение запускается в браузере.
2. Отображается фон 1000×600.
3. В центре экрана отображается анимированный объект размером не менее 400×400.
4. Анимация содержит минимум 5 кадров и воспроизводится циклически.
5. Объект плавно перемещается по клику или касанию.
6. Траектория движения визуально нелинейная.
7. Объект не выходит за границы игрового поля.
8. Быстрые повторные клики не ломают движение.
9. Несколько касаний обрабатываются предсказуемо.
10. При resize canvas масштабируется и центрируется.
11. На desktop размер canvas не превышает 1280×768.
12. На мобильных устройствах canvas занимает максимально возможную область.
13. Приложение пытается перейти в fullscreen после первого пользовательского жеста.
14. При отсутствии fullscreen API приложение использует fallback.
15. Код разделён на классы с понятной ответственностью.
16. Настраиваемые параметры вынесены в конфигурационные файлы.
17. В проекте есть README с описанием архитектуры, запуском и известными ограничениями.

## Учебная цель

Проект реализуется не только как тестовое задание, но и как учебный проект.

При разработке важно понимать:

* почему выбрана именно такая архитектура;
* как Phaser управляет сценами, ассетами, input и tween;
* как TypeScript помогает описывать игровые сущности;
* почему логика разделяется на классы;
* как canvas масштабируется в браузере;
* почему fullscreen требует fallback;
* как обрабатывать краевые случаи без усложнения кода.

Claude Code должен помогать пользователю писать проект самостоятельно: объяснять шаги, давать псевдокод, проверять решения и показывать альтернативы, но не писать весь проект целиком.
